Core Architecture: Sales Pipeline State Model

This shapes the entire database design. Pick the model that fits best.

A

Simple Status Field

Each lead has a single status enum field that gets updated as it moves through the pipeline. Fast to query, easy to filter.

Lead
status = ENUM(discovered, intro_emailed, intro_called,
            pricing_emailed, meeting_set, closed_won, closed_lost)
last_action_at = DateTimeField

Pros

  • Simple queries
  • Easy dashboard filters
  • Fast to build

Cons

  • No history of past states
  • Can't reconstruct what happened
  • Hard to show journey flowchart
B

Event-Sourced Pipeline

No status field on the Lead. Instead, every action is recorded as an immutable Event. The current state is derived by replaying events.

LeadEvent
lead = FK(Lead)
event_type = ENUM(intro_email_sent, call_made, pricing_sent, ...)
payload = JSONField  # email body, call notes, etc.
created_at = DateTimeField

Pros

  • Full audit trail
  • Perfect for flow chart
  • Never lose any data

Cons

  • Complex state derivation
  • Slower dashboard queries
  • Overkill for internal tool
C ★ Recommended

Stage + Action Log Hybrid

★ My recommendation — best of both worlds

Lead has a stage field for instant dashboard queries. Every action is also logged to LeadAction for history, call notes, email records, and the journey flowchart.

Lead
stage = ENUM(discovered → intro_sent → pricing_sent → meeting_set → closed)
auto_flow_paused = BooleanField  # the "stop flow" flag

LeadAction
lead = FK(Lead)
action_type = ENUM(intro_email, follow_up_call, pricing_email, meeting, note)
performed_by = FK(User)  # or "system" for automated
notes = TextField
metadata = JSONField  # email body, API response, etc.
created_at = DateTimeField

Pros

  • Fast dashboard queries (stage field)
  • Full history for flow chart
  • Stores call notes + email content
  • Simple to implement

Cons

  • Stage can drift if not updated carefully
  • Two places to keep in sync
